1. WHAT NAMEBLOOM IS
------------------------------------------------------------------------
A couples' baby-name-swiping web app. Two partners each swipe left/right
(Tinder-style) on baby names. When BOTH like the same name, it becomes a
"match" (with confetti). Goal: find names you both love, together.

Tech stack: Vite + React (JavaScript, .jsx). Deploy target: Netlify for
the static frontend + Supabase (Postgres + Auth + Realtime) for real
two-account / two-device sync.

KEY ARCHITECTURE DECISION — dual backend, one interface:
The app runs in one of two modes, chosen automatically:
  - DEMO MODE: if the Supabase env vars are absent. Uses browser
    localStorage. One person can simulate both partners on one device via
    a floating "switch partner" control. No real accounts, no sync.
  - SUPABASE MODE: if VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY are
    present at build time. Real accounts, real two-device sync.
Both modes implement the SAME store interface (src/lib/demoStore.js and
src/lib/supabaseStore.js), so screens never branch on backend. The picker
is src/lib/api.js.

NOTE: The password field + "show password" eye icon only appear in
Supabase mode (demo mode has no password). So if the deployed site shows
the password field, the env vars WERE present at build; if it doesn't,
they weren't.

2. FEATURES BUILT (all implemented and working)
------------------------------------------------------------------------
- Account creation + sign-in (Supabase Auth in real mode).
- Partner linking by email (mutual link).
- Projects (e.g. "My first baby"): pick which of 27 cultures to include,
  star favorites (favorites surface ~2:1 in the deck), gender filter
  (Surprise us / Boy / Girl / Unisex), optional surname for full-name
  preview, and a match goal (default target).
- Swipe deck: draggable cards with live "Love it / Pass / Top pick"
  stamps; super-like (swipe up or the star button); keyboard arrows
  (left = pass, up = top pick, right = love).
- Each card shows: name, phonetic pronunciation, meaning, origin/culture,
  full-name (surname) preview, "also spelled" alternatives, a "your
  partner loved this" ribbon, and a tap-to-hear speaker button (uses the
  browser SpeechSynthesis API).
- Matches screen: names both partners liked. Now GROUPED into
  "Top picks" (anything either partner super-liked, shown first with a
  gold highlight) and "More matches" below. Also: finalists (when BOTH
  partners star a name it becomes a mutual "Finalist"), vetoes (remove a
  match), share/export the shortlist, confetti on new matches, and a
  progress bar to the match goal.
- Realtime: in Supabase mode, swipes/matches sync live across devices.

3. THE NAMES DATASET
------------------------------------------------------------------------
~1,405 curated names across 27 cultures (roughly 45-85 per culture).
Each name has: gender (boy/girl/unisex), culture, origin label, meaning,
phonetic pronunciation (US-friendly respelling, e.g. "shih-VAWN"), and
(for ~107 names) common spelling alternatives.

Cultures: English, Irish & Celtic, Scottish, French, Italian,
Spanish & Latin, German, Scandinavian, Slavic, Greek, Armenian, Hebrew,
Arabic, Persian, Turkish, Indian & Sanskrit, Japanese, Chinese, Korean,
African, Hawaiian, American (modern), Russian, Polish,
Portuguese & Brazilian, Vietnamese, Filipino.

HOW THE DATA IS GENERATED (important for future edits):
- scripts/generate_names.mjs holds the name+meaning lists per culture and
  writes public/names.json.
- scripts/pron_data.mjs holds two maps: PRON (name -> pronunciation) and
  ALTS (name -> alternate spellings). The generator merges these in.
- To regenerate after editing: run  npm run gen:names
- Honest caveat: pronunciations are respellings (not IPA); accurate for
  standard pronunciation but names vary by region/family. The dataset is
  curated, not exhaustive. To grow it: add entries to the generator, or
  import a CSV into Supabase.

6. DIAGNOSTICS BUILT INTO THE APP (to help debug the connection)
------------------------------------------------------------------------
src/lib/supabase.js now:
  - Validates the URL shape and flags obvious mistakes (dashboard URL
    used instead of API URL, URL and key swapped, malformed URL).
  - Logs the resolved URL to the browser console on load:
      [NameBloom] Supabase URL in use: <url> | anon key: <masked>
  - Exports testSupabaseConnection() which pings the GoTrue health
    endpoint and returns plain-English status.
src/screens/Auth.jsx runs that self-test on load and shows, on the
sign-in screen:
  - "Checking connection to your server..." while testing
  - green "Connected to Supabase. You can create an account." on success
  - a specific red reason on failure (can't reach it / returned a web
    page not the API / anon key rejected / paused project, etc.)
supabaseStore.js signUp/signIn now convert low-level fetch/JSON failures
into a human message instead of the raw "Unexpected token '<'" error.

So once a correct build is live, the sign-in screen itself will tell you
whether Supabase is wired up correctly.
